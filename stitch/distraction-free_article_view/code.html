Designing a Real-Time Analytics Pipeline for Scale

Sarah Engineer verified
Principal Data Engineer @ TechCorp
Nov 14, 2024 schedule 15 min read

When designing data pipelines that need to ingest millions of events per second, the margin for error is razor-thin. In this technical deep dive, we'll explore how to architect a robust real-time analytics system using Kafka, Flink, and ClickHouse. This is based on a real-world system design interview question I encountered at a FAANG company.

The core challenge lies in handling backpressure while maintaining exactly-once processing semantics. Let's break down the data flow before diving into the implementation details.

"A standard lambda architecture is often too complex to maintain. Instead, we propose a Kappa architecture where all data flows through a single stream processing layer."

Ingestion Layer Implementation

For the ingestion layer, we use Go due to its excellent concurrency primitives. The goal is to accept JSON payloads, validate them, and push to Kafka.

GO producer.go 

func ProduceEvent(ctx context.Context, topic string, event Event) error {
    payload, err := json.Marshal(event)
    if err != nil {
        return fmt.Errorf("failed to marshal event: %w", err)
    }
    msg := &kafka.Message{
        TopicPartition: kafka.TopicPartition{Topic: &topic, Partition: kafka.PartitionAny},
        Value:          payload,
        Key:            []byte(event.UserID), // Ensure ordering by UserID
    }
    return producer.Produce(msg, nil)
}

Processing Logic

Once data is in Kafka, we need to process it. Python is excellent for defining the transformation logic, especially with libraries like PyFlink or Faust.

PY processor.py 

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment
def process_stream():
    env = StreamExecutionEnvironment.get_execution_environment()
    t_env = StreamTableEnvironment.create(env)
    # Define source table from Kafka
    t_env.execute_sql("""
        CREATE TABLE source_events (
            user_id STRING,
            event_type STRING,
            ts TIMESTAMP(3),
            WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
        ) WITH (
            'connector' = 'kafka',
            'topic' = 'raw_events',
            'format' = 'json'
        )
    """)

Storage & Query Strategy

Finally, we persist the aggregated data into ClickHouse. The schema design is crucial for query performance. We use a `MergeTree` engine partitioned by date.

SQL clickhouse_schema.sql 

CREATE TABLE analytics.user_activity (
    timestamp DateTime,
    user_id UInt64,
    event_type LowCardinality(String),
    duration_ms UInt32,
    meta_data Map(String, String)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(timestamp)
ORDER BY (event_type, timestamp, user_id)
TTL timestamp + INTERVAL 1 YEAR;

Conclusion

Designing scalable systems is about understanding trade-offs. By decoupling ingestion from processing and choosing the right storage engine, we achieved a throughput of 1M events/sec with sub-second latency.
